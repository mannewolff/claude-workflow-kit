## Kontext
Hauptverbraucher des GraphQL-Kontingents: GitHubIssueTracker._getProjectItemId in kit/board.mjs laedt bei jedem moveIssue die komplette Projekt-Item-Liste (gh project item-list --limit 1000, intern paginiert zu 100/Seite mit verschachtelten Feldern — je nach Boardgroesse 20-100+ Punkte pro Aufruf). createIssue ruft moveIssue wegen Eventual Consistency bis zu 3x im Retry auf. Eine /issues-Session mit anschliessendem /implement-ready kann so 1.000-3.000 Punkte verbrauchen.

Alternative seit September 2025: GitHub bietet eine REST-API fuer Projects v2 (/users/{owner}/projectsV2/{number}/items bzw. /orgs/...), die Items inkl. Feldwerten liefert und Single-Select-Felder per PATCH setzen kann — mit eigenem, getrenntem Kontingent (REST: 5.000 Requests/h).

Hinweis: kit/board.mjs wird mit dem board-ui-Repo gesynct — Aenderung dort nachziehen.

## Aufgabe
In kit/board.mjs, Klasse GitHubIssueTracker:
- _getProjectItemId ersetzen durch einen gezielten Lookup pro Issue statt Voll-Liste. Bevorzugt via GraphQL-Einzelabfrage (repository -> issue(number: N) -> projectItems(first: 10), gefiltert auf die konfigurierte projectNumber; Kosten ~1 Punkt) ODER via Projects-v2-REST-Endpoints, falls damit auch moveIssue (PATCH Single-Select) umgestellt wird. Entscheidung im Issue-Kommentar dokumentieren.
- listIssues(status) bleibt in diesem Issue unveraendert (Voll-Liste dort weiterhin noetig).
- Der Retry-Loop in createIssue (Eventual Consistency) bleibt bestehen, profitiert aber automatisch vom guenstigeren Lookup.

## Akzeptanzkriterium
- Ein moveIssue auf einem Board mit mehr als 100 Items verursacht keine paginierte item-list-Abfrage mehr.
- Vorher/Nachher-Messung via `gh api rate_limit` dokumentiert: Punkteverbrauch pro moveIssue nachher maximal ~5 (vorher je nach Boardgroesse 20-100+).
- Fehlerfall bleibt verstaendlich: moveIssue fuer ein Issue, das nicht im Projekt haengt, liefert weiterhin einen BoardError mit Issue-Nummer und Projekt-Nummer (kein Stacktrace).
- board.mjs bleibt als Single-File-Tool lauffaehig ohne weiteren Repo-Kontext.

## Abhängigkeiten
Keine. (Beruehrt denselben Codebereich wie das Issue "Projekt-Metadaten persistent cachen" — Merge-Reihenfolge beliebig, aber nacheinander umsetzen.)

### Hinweise
- **Entscheidung GraphQL statt REST:** Bewusst bei einer gezielten GraphQL-Einzelabfrage geblieben statt Umstieg auf die Projects-v2-REST-API. Grund: minimaler, lokal begrenzter Eingriff (nur _getProjectItemId), keine Umstellung von moveIssue/PATCH nötig, bleibt im vorhandenen `gh`-Werkzeug. Die 3-Punkte-Kosten lösen das Kontingentproblem bereits vollständig; ein REST-Vollumstieg (eigener Kontingent-Topf) bleibt als separater, größerer Schritt möglich.
- **Sync-Repo:** `kit/board.mjs` wird mit dem board-ui-Repo gesynct — Änderung dort nachziehen.
- Board #11 hat aktuell 71 Items (<100); das Akzeptanzkriterium nannte '>100', der entscheidende Nachweis (kein item-list, konstante ~3 Punkte) ist größenunabhängig erbracht.
- Zusammenspiel mit #0001: Der Selbstheilungs-Pfad (moveIssue → item-edit-Fehler → Cache-Reload → Retry) nutzt weiterhin diesen Lookup und wurde im #0001-Test grün durchlaufen.
