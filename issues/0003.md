## Kontext
Das Kit braucht eine sichtbare Versionskennung im Format x.y.z, die sich bei `push main` (z) und `merge production` (y) automatisch erhoeht; x nur auf explizite Ansage. Start: 1.4.0. Voraussetzung ist ein zuverlaessiger, skriptbarer Bump-Mechanismus und ein Speicherort, den die Board-UI lesen kann. board-ui.mjs liest bereits `workflow.config.json` (via /api/config) — daher ist ein `version`-Feld dort der integrierte Ort (keine separate VERSION-Datei). Semver-Standard: ein y-Bump setzt z auf 0, ein x-Bump setzt y und z auf 0.

## Aufgabe
- `version` in `.claude/workflow.config.json` einfuehren, Startwert "1.4.0". Ebenso in `templates/workflow.config.json` (Default "1.4.0") und in `templates/workflow.config.schema.json` (Feld dokumentieren, Format x.y.z).
- Neues Single-File-Tool `tools/version.mjs` mit Flags:
  - `--patch` → z+1
  - `--minor` → y+1, z=0
  - `--major` → x+1, y=0, z=0
  - `--get` → aktuelle Version auf stdout ausgeben, ohne zu aendern (fuer Anzeige/Debug)
- Liest/schreibt das `version`-Feld der Config ueber dieselbe Kandidaten-Aufloesung wie board.mjs (`.claude/workflow.config.json`, dann `workflow.config.json`, relativ zum Arbeitsverzeichnis). Gibt nach einem Bump die neue Version auf stdout aus.
- Validiert das bestehende Format strikt als drei nicht-negative Ganzzahlen x.y.z; bei Ungueltigkeit Fehlermeldung auf stderr + Exit 1, Config unveraendert.

## Akzeptanzkriterium
- `node tools/version.mjs --get` gibt bei Startzustand "1.4.0" aus.
- `--patch`: 1.4.0 → 1.4.1, erneut → 1.4.2.
- `--minor`: 1.4.2 → 1.5.0 (z auf 0).
- `--major`: 1.5.0 → 2.0.0 (y und z auf 0).
- Ungueltiger Bestand (z.B. "1.4" oder fehlendes Feld) → Fehlermeldung + Exit 1, Config bleibt unveraendert.
- `tools/version.mjs` ist lauffaehig ohne weiteren Repo-Kontext (Single-File, nur `node:*`-Imports), Config-Pfad relativ zum Arbeitsverzeichnis aufgeloest.

## Abhängigkeiten

### Hinweise
- **Scope-Annahme:** Versionskennung = die des **Kit-Repos selbst** (gekoppelt an die kit-spezifische merge-production → docs.mwolff.org-Pipeline). Daher ist `tools/version.mjs` bewusst kit-lokal und wird nicht in Zielprojekte verteilt. Falls doch eine Pro-Projekt-Versionierung gewünscht ist, müsste die Bump-Logik in das verteilte board.mjs wandern — bitte vor #0004-Abnahme klären.
- **Board-Anzeige** (config.version rendern) ist bewusst kein Kit-Issue → gehört ins board-ui-Repo; dort das Feld optional behandeln (fehlt es, nichts anzeigen), damit fremde Projekte ohne version-Feld nicht brechen.
- Die echte Kit-Config bleibt bei 1.4.0 — der erste reale Bump passiert erst durch den nächsten push main (nach #0004).
