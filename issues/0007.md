## Kontext

Beim Test von implement-ready wurde #1 ueber node .claude/kit/board.mjs issue move 1 in_progress verschoben. Der Adapter meldete { ok: true, status: 'in_progress' }, danach hatte das Issue aber gar kein Label mehr — statt "In Progress". Damit rutscht das Issue label-los in die Open-Spalte statt in die Zielspalte; der Move ist faktisch fehlgeschlagen.
Ursachen-Nachweis: glab issue update 1 --label \"In Progress\" allein setzt das Label korrekt. Das Problem liegt also im kombinierten Update-Aufruf von moveIssue().
Root Cause: GitLabIssueTracker.moveIssue() (aktuell Zeile 524-527) baut das --unlabel-Set aus allen Spalten-Labels — inklusive der Ziel-Spalte — und haengt danach --label <Ziel> an denselben glab issue update-Aufruf. glab verrechnet --unlabel 'In Progress' und --label 'In Progress' gegeneinander, sodass das Ziel-Label netto nicht gesetzt wird.

## Aufgabe



        
      Datei: .claude/kit/board.mjs, Methode GitLabIssueTracker.moveIssue().

        
      Das Ziel-Label aus dem --unlabel-Set ausnehmen:


const unlabelArgs = Object.values(labels)
  .filter((l) => l !== label)
  .map((l) => `--unlabel ${shellQuote(l)}`)
  .join(' ');



Verwandt mit #3: Fuer die Spalten backlog (-> Open) und done (-> Closed) ist "Open"/"Closed" kein Label, sondern GitLab-Zustand. Der Move dorthin darf kein gleichnamiges Label setzen, sondern muss nur die Status-Labels entfernen bzw. das Issue schliessen. Beim Fix mitdenken, damit move nach backlog/done nicht erneut ein Phantom-Label erzeugt.

## Akzeptanzkriterium



        
      
node .claude/kit/board.mjs issue move <id> in_progress setzt genau das Label "In Progress" und entfernt die anderen Status-Labels; das Issue erscheint in der In-Progress-Spalte.

        
      Gleiches fuer ready und in_review.

        
      
... move <id> backlog laesst das Issue offen und ohne Status-Label (kein Phantom-Label).

        
      
... move <id> done schliesst das Issue (kein Phantom-Label "Closed").

        
      Nach jedem Move zeigt glab issue view <id> exakt das erwartete Label bzw. den erwarteten Zustand.

## Abhaengigkeiten

Keine. Gleicher Adapter wie #2 und #3; #3 und dieses Issue ueberschneiden sich bei der Sonderbehandlung von backlog/done und sollten zusammen umgesetzt werden.
