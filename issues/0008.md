## Kontext

Der GitLab-Adapter behandelt die Backlog-Spalte an zwei Stellen fest verdrahtet, statt dem Config-Mapping columns.backlog zu folgen. Dadurch widersprechen sich die Operationen, sobald ein Projekt backlog als echtes Label fuehrt (Variante A: columns.backlog = \"Backlog\", Board hat eine label-basierte "Backlog"-Liste):


        
      
createIssue() setzt hart glab issue update <id> --label \"Backlog\" (aktuell Zeile 479) und ignoriert das Mapping.

        
      
moveIssue() behandelt to === \"backlog\" in der Open/Closed-Sonderbehandlung (aktuell Zeile 525): es entfernt alle Status-Labels und reopnet, setzt also kein Backlog-Label. Ein Move nach backlog wirft das Issue damit aus der Backlog-Spalte.

Folge: Anlegen landet zufaellig richtig (weil createIssue "Backlog" hartkodiert), aber move <id> backlog und list --status backlog passen nicht dazu. createIssue, moveIssue, listIssues und labelToStatus muessen dieselbe Quelle der Wahrheit benutzen: das Config-Mapping.
Vorgeschichte: Bugs #2 (list --state), #3 (createIssue Phantom-Label) und #4 (moveIssue verliert Ziel-Label) wurden bereits adressiert; dieses Issue schliesst die verbliebene Inkonsistenz bei der backlog-Spalte.
## Aufgabe



        
      Datei: .claude/kit/board.mjs, Klasse GitLabIssueTracker.

        
      
createIssue(): das hartkodierte --label \"Backlog\" durch columnLabels(cfg).backlog ersetzen. Sonderfall beachten: mappt backlog auf den nativen Open-Zustand (kein echtes Label), darf kein Label gesetzt werden — dann Issue einfach offen lassen.

        
      
moveIssue(): \"backlog\" aus der Open/Closed-Sonderbehandlung herausnehmen, wenn backlog per Config ein echtes Label ist, sodass backlog den normalen Label-Pfad laeuft (Ziel-Label setzen, andere Status-Labels entfernen — Ziel-Label nicht im selben Aufruf unlabeln, vgl. #4). \"done\" bleibt als Closed-Zustand.

        
      Entscheidungskriterium einheitlich definieren: Wann ist eine Spalte ein GitLab-Zustand (Open/Closed) und wann ein Label? Vorschlag: Zustaende sind ausschliesslich Open (backlog, falls so gemappt) und Closed (done); alle anderen Mappings sind Labels. Diese Regel an einer Stelle kapseln und von create/move/list/labelToStatus nutzen.

        
      Falls das Kit eine Default-/Template-workflow.config.json ausliefert: deren columns.backlog-Default konsistent zur gewaehlten Semantik setzen und dokumentieren.

## Akzeptanzkriterium



        
      Mit columns.backlog = \"Backlog\": issue create legt ein Issue mit Label "Backlog" an, das in der Backlog-Spalte erscheint; issue move <id> backlog setzt/behaelt das Label "Backlog"; issue list --status backlog findet es; labelToStatus gibt backlog zurueck.

        
      Mit columns.backlog = \"Open\" (Open-Zustand): issue create legt ein offenes Issue ohne Status-Label an; issue move <id> backlog entfernt die Status-Labels und laesst das Issue offen; kein Phantom-Label "Open".

        
      
issue move <id> done schliesst das Issue in beiden Varianten (kein Phantom-Label "Closed").

        
      Kein Vorkommen von hartkodiertem "Backlog" mehr im GitLab-Adapter.
