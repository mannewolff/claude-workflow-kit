## Kontext

`install.mjs` wirbt im Kopf-Kommentar mit "Single-File-Portabilitaet", haelt dieses
Versprechen aber nur teilweise ein: `CLAUDE_WORKFLOW_MD` ist als Template-String
eingebettet, `kit/board.mjs` und `kit/board-ui.mjs` sind als Base64-Blobs eingebettet
(`BOARD_MJS_B64`, `BOARD_UI_MJS_B64`, gepflegt ueber `tools/sync-blobs.mjs`). Die
zehn Skills dagegen sind **nicht** eingebettet: `copySkills()` liest sie ueber
`skillsSrc = join(__dirname, "skills")` live vom Dateisystem neben `install.mjs` —
das funktioniert nur, wenn das komplette Repo vorliegt (z. B. per `git clone`).

`docs.mwolff.org/install.mjs` wird jedoch als **einzelne Datei** ausgeliefert:
`tools/copy-install-for-docs.mjs` kopiert nur `install.mjs` nach `docs/public/`,
ohne begleitenden `skills/`-Ordner. Wer die Datei von dort einzeln herunterlaedt
(z. B. `curl -O https://docs.mwolff.org/install.mjs && node install.mjs`, der im
README dokumentierte Installationsweg), bekommt deshalb **garantiert** einen leeren
`skills/`-Zielordner — reproduzierbar bei jedem Download dieser Art, kein Einzelfall.

Verschaerft wird das durch einen zweiten Fehler in `copySkills()`: `mkdirSync(targetDir,
{ recursive: true })` laeuft unbedingt **vor** der Kopier-Schleife. Selbst wenn danach
jeder einzelne Skill mangels Quelle uebersprungen wird (nur eine leicht zu uebersehende
`console.warn`-Zeile pro Skill, kein Exit-Code ≠ 0), liegt am Ende ein scheinbar
"installierter", tatsaechlich aber leerer `skills/`-Ordner da — der Fehlschlag sieht
wie ein Erfolg aus.

Der in README dokumentierte Alternativweg `npx github:mannewolff/claude-workflow-kit`
waere von diesem Problem vermutlich nicht betroffen (voller Repo-Kontext), ist aber im
Code selbst als "(nach Veroeffentlichung)" markiert und mangels `package.json` mit
`bin`-Eintrag aktuell nicht funktionsfaehig — fuer die meisten Nutzer bleibt der defekte
Weg ueber docs.mwolff.org/install.mjs damit der einzig sichtbare.

Gefunden beim Versuch, das Kit in einem neuen Projekt (`shell-app`) einzurichten:
`.claude/kit/board.mjs`, `.claude/kit/board-ui.mjs` und `.claude/CLAUDE-workflow.md`
wurden korrekt angelegt, `.claude/skills/` existierte, war aber komplett leer — kein
Skill (`/plan`, `/issues`, `/implement-ready`, ...) war danach weder projekt- noch
global verfuegbar.

## Aufgabe

- `tools/sync-blobs.mjs` um die zehn Skill-Ordner erweitern — analog zu
  `BOARD_MJS_B64`/`BOARD_UI_MJS_B64` (Format offen: ein Blob pro Skill oder ein
  gemeinsames Tar/Zip als ein Blob, je nachdem was die Regex-basierte Ersetzung in
  `install.mjs` einfacher haelt).
- `install.mjs`: `copySkills()` so anpassen, dass sie primaer aus den eingebetteten
  Blobs schreibt (Verzeichnisse + Dateien dekodieren), statt zwingend vom Dateisystem
  neben `install.mjs` zu lesen. Ein Fallback aufs Dateisystem kann optional erhalten
  bleiben (z. B. fuer Entwicklung direkt im Kit-Repo).
- `copySkills()`: `mkdirSync(targetDir, ...)` nicht mehr bedingungslos vor der Schleife
  aufrufen — Zielordner erst anlegen, wenn tatsaechlich mindestens ein Skill kopiert
  wird. Bei komplettem Fehlschlag (keine Quelle/kein Blob gefunden): klare
  Fehlermeldung auf stderr **und** Exit-Code ≠ 0, statt nur einer Warnung pro Skill —
  ein fehlgeschlagener Install darf nicht wie ein Erfolg aussehen.
- `docs.mwolff.org/install.mjs` (ueber `copy-install-for-docs.mjs` bzw. beim naechsten
  Docs-Deploy) mit den eingebetteten Blobs neu veroeffentlichen.

## Akzeptanzkriterium

- Frisches Verzeichnis **ohne** jeglichen `claude-workflow-kit`-Repo-Kontext: nur
  `curl -O https://docs.mwolff.org/install.mjs && node install.mjs` ausgefuehrt →
  `.claude/skills/` enthaelt alle zehn Skill-Unterordner mit vollstaendigem Inhalt
  (Diff gegen `claude-workflow-kit/skills/` ist leer).
- Derselbe Test lokal aus dem geklonten Repo heraus (`node install.mjs`) liefert
  weiterhin identisches Ergebnis — keine Regression fuer den bisherigen Weg.
- Absichtlich simulierter Fehlerfall (Blob fehlt/korrupt): Installer bricht mit
  Exit-Code ≠ 0 und eindeutiger Fehlermeldung ab, statt klaglos einen leeren
  `skills/`-Ordner anzulegen.
- `node tools/sync-blobs.mjs --check` erkennt Drift, wenn ein Skill-Ordner geaendert
  wurde, ohne dass die Blobs in `install.mjs` neu generiert wurden (analoges Verhalten
  zum bestehenden Check fuer `kit/board.mjs`/`kit/board-ui.mjs`).

## Architektonische Entscheidungen

1. **Blob-Format: ein gemeinsamer JSON-Blob statt 10 Einzel-Konstanten.** Jeder Skill-Ordner enthaelt aktuell genau eine Datei (`SKILL.md`). Struktur `{ skillName: { dateiname: inhalt } }`, JSON-serialisiert, einmal Base64-kodiert (`const SKILLS_B64 = "..."` in install.mjs, analog `BOARD_MJS_B64`). Ein Regex-Ersetzungsblock in sync-blobs.mjs statt zehn. Verschachtelung nach Dateiname macht das Format robust, falls ein Skill spaeter weitere Dateien bekommt.

2. **sync-blobs.mjs generalisieren statt duplizieren.** BLOBS-Liste um eine Variante `{constName, sourceDir}` erweitern (rekursiv einlesen, JSON bauen, Base64), die denselben Vergleichs-/Ersetzungspfad durchlaeuft wie die bestehenden Einzeldatei-Eintraege. `--check` bleibt ein einziger Mechanismus fuer alle drei Blobs.

3. **copySkills(): Blob primaer, Dateisystem-Fallback pro Skill.** Pro Skill zuerst aus dem dekodierten SKILLS_B64 schreiben; nur wenn dort kein Eintrag existiert, auf skillsSrc/<skill> (Dateisystem) zurueckfallen (fuer Kit-Entwicklung im geklonten Repo). Zielverzeichnis nicht mehr pauschal vor der Schleife anlegen, sondern erst pro Skill unmittelbar vor dem Schreiben (mkdirSync(join(targetDir, skill), {recursive:true})) — ein Skill ohne Quelle hinterlaesst keinen leeren Ordner.

4. **Fehlerverhalten: hart nur bei Totalausfall.** Schlagen alle 10 Skills fehl (Blob leer/korrupt und kein Dateisystem-Fallback), bricht der Installer mit klarer stderr-Meldung und process.exit(1) ab. Schlaegt nur ein Teil fehl, bleibt es beim bisherigen Verhalten (Warnung pro Skill, Installation laeuft weiter) — deckt sich mit den in der Issue formulierten Akzeptanzkriterien.

## Geplante Aenderungen je Datei

**tools/sync-blobs.mjs**
- BLOBS-Eintrag `{ constName: "SKILLS_B64", sourceDir: join(root, "skills") }`
- Hilfsfunktion `buildSkillsJson(dir)`: liest alle Unterordner, pro Ordner alle Dateien flach, gibt `{skillName: {filename: content}}` zurueck
- Vergleichs-/Ersetzungslogik so anpassen, dass sie fuer sourceDir-Eintraege den JSON-Blob als Vergleichsbasis nimmt

**install.mjs**
- Neue Konstante `SKILLS_B64` (per sync-blobs.mjs gepflegt)
- copySkills(skillsSrc, targetDir) umbauen wie oben beschrieben; nach der Schleife bei 0 kopierten Skills: console.error + process.exit(1)
- Kopf-Kommentar (Zeile 2-6) aktualisieren, beschreibt aktuell die falsche Funktionsweise

**docs.mwolff.org/install.mjs** — keine separate Aktion; wird beim naechsten Merge nach production automatisch ueber den bestehenden prebuild-Hook (docs-site/package.json) + deploy-docs.yml neu ausgerollt.

**CI** — blob-sync-check.yml laeuft bereits generisch gegen sync-blobs.mjs --check, deckt den neuen Blob ohne Aenderung ab.

## Verifizierung

1. `node tools/sync-blobs.mjs` lokal → SKILLS_B64 generiert, `--check` danach gruen
2. Regressionstest: `node install.mjs` aus dem geklonten Repo → .claude/skills/ identisch zu skills/ (diff -r)
3. Kernszenario: install.mjs allein in leeres Temp-Verzeichnis kopieren (ohne skills/), `node install.mjs` ausfuehren → alle 10 Skill-Ordner vollstaendig, diff -r gegen skills/ leer
4. Fehlerfall: SKILLS_B64 temporaer auf "" setzen, im selben leeren Temp-Verzeichnis installieren → Exit-Code ≠ 0, eindeutige stderr-Meldung, kein leerer skills/-Ordner
5. Drift-Erkennung: eine SKILL.md lokal aendern ohne sync-blobs.mjs erneut laufen zu lassen → `--check` schlaegt fehl (Exit 1)

## Abschlussbericht Issue #0011

### Änderungen
- `tools/sync-blobs.mjs` — BLOBS-Eintraege unterstuetzen jetzt `sourceDir` (Ordner statt Einzeldatei); neue Hilfsfunktion `buildDirJson()` liest `skills/` rekursiv zu `{skillName: {dateiname: inhalt}}` und base64-kodiert das Ergebnis als `SKILLS_B64`.
- `install.mjs` — neue Konstante `SKILLS_B64` (per sync-blobs.mjs gepflegt); `copySkills()` umgebaut: liest primaer aus dem eingebetteten Blob, faellt pro Skill nur fuer die Kit-Entwicklung im geklonten Repo aufs Dateisystem zurueck; Zielordner wird erst pro Skill vor dem Schreiben angelegt statt pauschal vorab; bei komplettem Fehlschlag (0 kopierte Skills) `console.error` + `process.exit(1)` statt stiller Warnungen.

### Tests und Checks
- `node tools/sync-blobs.mjs --check` → gruen (Blobs synchron)
- Regressionstest `node install.mjs` im geklonten Repo → `.claude/skills/` identisch zu `skills/` (`diff -r` leer)
- Standalone-Test: `install.mjs` allein in leerem Verzeichnis ohne begleitenden `skills/`-Ordner → alle zehn Skill-Ordner vollstaendig aus dem Blob geschrieben, `diff -r` gegen `skills/` leer
- Fehlerfall simuliert (`SKILLS_B64=""`, kein Dateisystem-Fallback) → Exit-Code 1, eindeutige stderr-Meldung, kein `.claude/skills/`-Ordner angelegt
- Drift-Erkennung: `skills/plan/SKILL.md` testweise geaendert ohne `sync-blobs.mjs` neu laufen zu lassen → `--check` schlaegt mit Exit 1 fehl; nach Rueckaenderung wieder gruen (bis auf vorbestehende, nicht zu diesem Issue gehoerende Drift in `kit/board-ui.mjs`, siehe Hinweise)
