## Kontext

`GitHubIssueTracker._projectNumber()` (kit/board.mjs) wirft einen klaren `BoardError`, wenn `github.projectNumber` in `workflow.config.json` fehlt. `listIssues` faengt das bereits ab und degradiert sanft auf eine ungefilterte Liste offener Issues. `moveIssue` tut das nicht: es ruft `_ensureProjectMeta()` unbedingt auf, das wiederum `_projectNumber()` braucht — ohne die Config-Nummer bricht jede Board-Bewegung hart ab. In der Praxis fuehrt das dazu, dass Board-Bewegungen ersatzweise manuell per `gh` direkt gegen das GitHub Project gemacht werden muessen, statt ueber den Adapter zu laufen. `createIssue` haengt das Issue mangels `github.projectNumber` still gar nicht erst ans Board — ohne Fehlermeldung oder Hinweis.

Fuer die meisten Projekte gibt es pro Owner genau ein GitHub Project — die Nummer laesst sich in diesem Fall automatisch ermitteln (`gh project list --owner <owner>`), ohne dass ein Mensch sie manuell in die Config eintragen muss.

## Aufgabe

In `kit/board.mjs`, Klasse `GitHubIssueTracker`:

- `_projectNumber()`: wenn `this._cfg.github?.projectNumber` fehlt, `gh project list --owner <owner> --format json` aufrufen.
  - Genau 1 Treffer: dessen Nummer verwenden, per `process.stderr.write` einen Hinweis ausgeben ("automatisch erkanntes einziges GitHub Project #N ('Titel') fuer Owner 'X' wird verwendet. Zur dauerhaften Fixierung ergaenzen: '\"github\": { \"projectNumber\": N }'"). Kein automatisches Schreiben in `workflow.config.json`.
  - 0 Treffer: `BoardError` ("kein GitHub Project fuer Owner gefunden").
  - ≥2 Treffer: `BoardError` mit einer Liste aller gefundenen Projects (`#N (Titel)`, komma-getrennt).
  - Ergebnis pro Prozess in einem Instanzfeld memoisieren (analog `_repoName`/`_projectId`), damit `_projectNumber()` innerhalb eines `board.mjs`-Aufrufs nur einmal aufloest.
- Auto-erkannte Nummer zusaetzlich prozessuebergreifend cachen: neue private Helper `_readAutoProjectNumberCache(owner)` / `_writeAutoProjectNumberCache(owner, num)`, die dieselbe Cache-Datei wie `_readMetaCache()`/`_writeMetaCache()` (`.claude/board-meta-cache.json`) unter einem eigenen Schluessel `${owner}#auto` nutzen. Damit kostet nicht jeder Aufruf ohne `github.projectNumber` erneut `gh project list` (Fortsetzung der Kontingent-Sparsamkeit aus Issue #0001).
- `listIssues` (Status-Filter-Zweig): den bisherigen Guard `if (!this._cfg.github?.projectNumber) { ... }` durch `try { num = this._projectNumber() } catch { ...gleicher Hinweistext auf stderr..., return this.listIssues(undefined) }` ersetzen. Verhalten bei 0/≥2 Projects bleibt sanft (ungefilterte Liste), gewinnt aber den Auto-Detect-Fall bei genau 1 Project hinzu.
- `createIssue`: den aeusseren Guard `if (this._cfg.github?.projectNumber) { ... }` entfernen. Der bestehende try/catch um die Board-Zuordnung faengt eine `BoardError` aus `_projectNumber()` bereits ab und gibt `"Hinweis: Board-Zuordnung fehlgeschlagen: ..."` aus — dadurch werden neue Issues bei genau einem Project automatisch ans Board gehaengt, statt die Zuordnung mangels konfigurierter Nummer still zu ueberspringen.
- `moveIssue`: keine Codeaenderung noetig, profitiert automatisch von der neuen `_projectNumber()`.
- `node tools/sync-blobs.mjs` ausfuehren, damit `BOARD_MJS_B64` in `install.mjs` mit dem geaenderten `kit/board.mjs` synchron ist.
- `docs/dokumentation.md` (Abschnitt zu `github.projectNumber`, aktuell Zeile ~119) und der Kopf-Kommentar in `kit/board.mjs` (Zeile ~70, "Fuer GitHub-Board-Integration: github.projectNumber in der Config setzen.") um das neue Fallback-Verhalten ergaenzen.

## Akzeptanzkriterium

- Test-Repo mit `issueTracker: github`, `github.projectNumber` NICHT in der Config, Owner hat genau ein GitHub Project: `node .claude/kit/board.mjs issue list --status ready` und `node .claude/kit/board.mjs issue move <id> <status>` funktionieren beide (kein `BoardError`), auf stderr erscheint der Auto-Erkennungs-Hinweis.
- Zweiter Aufruf direkt danach: kein erneuter `gh project list`-Call zur Auto-Erkennung (nachweisbar z. B. via `GH_DEBUG=api`) — die Nummer kommt aus dem Cache in `.claude/board-meta-cache.json`.
- Gleiches Test-Repo mit einem zweiten, kuenstlich angelegten GitHub Project (also 2 Projects fuer den Owner) und weiterhin ohne `github.projectNumber`: `issue move` bricht mit `BoardError` ab, die Meldung listet beide gefundenen Projects (Nummer + Titel) auf. `issue list --status ready` degradiert weiterhin sanft auf eine ungefilterte Liste offener Issues.
- Test-Repo ganz ohne GitHub Project fuer den Owner, ohne `github.projectNumber`: `issue move` bricht mit `BoardError` ab ("kein GitHub Project gefunden").
- `createIssue` in einem Test-Repo mit genau einem Project und ohne `github.projectNumber` in der Config: neu angelegtes Issue landet automatisch auf dem Board (Status backlog), kein manueller Zusatzschritt noetig.
- Regressionstest: Test-Repo MIT explizit gesetztem `github.projectNumber` verhaelt sich unveraendert (kein zusaetzlicher `gh project list`-Call zur Auto-Erkennung).
- `node tools/sync-blobs.mjs --check` laeuft nach der Aenderung gruen.
- Portabilitaet: `kit/board.mjs` bleibt lauffaehig ohne weiteren Repo-Kontext — ein isoliert (z. B. via `install.mjs` in ein frisches Projekt) kopiertes `.claude/kit/board.mjs` verhaelt sich identisch zum Original im Kit-Repo (kein zusaetzlicher Import, keine neue Abhaengigkeit ausserhalb von `node:*` und der `gh`-CLI).
